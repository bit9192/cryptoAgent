# modules/context-engine 模块开发计划

目标：为 ex:lon 提供可持续任务的统一上下文读模型，覆盖 wallet tree、chat 摘要、env 快照与任务状态机；严格遵循先方案与样本、再实现、再验证的闭环流程。

## 1. 需求范围

本模块当前需求：

1. 统一上下文读模型
- 维护 session 级别上下文树
- 统一读取 wallet/chat/env/task 状态

2. 可持续任务状态管理
- 任务生命周期状态机
- checkpoint 与中断恢复语义

3. 安全字段分层
- public/internal/private/secret 输出分级
- AI/UI 对外输出自动脱敏

4. 事件驱动更新
- 通过 command/event 更新上下文树
- 支持最小化重放恢复

测试数据来源：

- 使用本目录 context-test-data.md 的样本清单
- 使用 src/test/TEST_DATA_RULES.md 的四类样本要求

## 2. 分层落位

严格按以下分层开发：

1. modules/context-engine
- 放置上下文模型、状态机、事件投影、字段过滤
- 不做 CLI 输入解析与 task 参数校验

2. execute
- 负责任务执行编排、checkpoint 触发、恢复流程接入
- 不承载上下文核心数据结构实现

3. tasks
- 只做动作分发与参数校验
- 通过 context-engine 的接口读写上下文

4. cli
- 只做 ex:lon 展示与交互路由
- 不写核心状态转移逻辑

## 3. 统一开发流程

每次新增或修改本模块能力，必须按以下顺序执行：

1. 先补充需求说明
- 明确本次只解决什么
- 明确本次不解决什么
- 明确输入、输出、边界、失败路径、恢复路径

2. 先准备测试数据
- 在本目录补 happy/edge/invalid/security 样本
- 未准备样本前，不进入实现

3. 先写切片计划
- 把切片写入“当前切片计划”
- 明确函数落点、测试文件、验收标准

4. 再开发实现
- 严禁在 task/cli 散落核心状态逻辑
- 严禁跨切片混做

5. 最后测试验证
- 先跑切片测试
- 再跑受影响模块测试

## 4. 当前模块设计原则

1. 统一读模型，分域写模型
- 读取可聚合，写入必须按域进行

2. 事件优先
- 变更先记录事件，再更新投影树

3. 可恢复优先
- 中断后可从 checkpoint 恢复

4. 安全优先
- secret 不落日志、不落快照、不出 AI/UI 输出

5. 有界缓存
- chat 原文保留窗口，长期存摘要

本次只做：

1. 定义任务状态流转规则
2. 定义 checkpoint 读写契约
3. 定义恢复失败降级规则

本次不做：

1. 多任务并发调度优化
2. 跨进程分布式锁
3. 历史回放 UI

验收标准：

1. 状态流转 happy/invalid 样本覆盖
2. checkpoint 恢复样本覆盖

2. env 脱敏快照写入上下文

本次不做：

1. 全量历史会话语义检索
2. env secret 原文保存
3. 高级告警策略

验收标准：

1. chat/env 可在统一读模型中读取
2. security 样本验证无 secret 泄露
3. 旧测试不回归
